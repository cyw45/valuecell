# 本机与 VPS 环境配置说明

本项目有两套运行时环境文件，不能混用，也不要提交到 Git。

> **禁止同步规则**：`%APPDATA%\ValueCell\.env` 与
> `docker/runtime/.env` 是不同机器、不同网络命名空间的配置文件。不得用
> `scp`、同步盘或 Git 将其中一份覆盖另一份。只同步模板与源码，每台机器在
> 本地填写自己的运行时 `.env`。

## 0. 立刻纠正已同步的 VPS 配置

如果已将 Windows 本机 `.env` 直接复制到 VPS，VPS 容器很可能正在使用公网
PostgreSQL 地址。请在 VPS 上编辑 `docker/runtime/.env`，将数据库地址替换为：

```env
VALUECELL_DATABASE_URL=postgresql+psycopg2://用户名:密码@host.docker.internal:5432/数据库名
```

不要复制 Windows 文件中的 `VALUECELL_DATABASE_URL`。VPS 修改后执行：

```bash
docker compose -f docker-compose.local.yml up -d --force-recreate backend
```

然后按第 4 节的容器内检查命令确认生效。

### 仅允许同步的内容

| 可以同步到 VPS | 不可同步到 VPS |
| --- | --- |
| 源码、`docker-compose.local.yml`、`docker/runtime/.env.example` | `%APPDATA%\ValueCell\.env` |
| 运行时变量名与注释 | `docker/runtime/.env`（除非在 VPS 本机手动编辑） |
| 不含密钥的 provider 顺序、刷新间隔 | 数据库 DSN、密码、JWT Secret、加密主密钥 |

正确流程是：在 VPS 从 `docker/runtime/.env.example` 创建
`docker/runtime/.env`，只在 VPS 上填入数据库和密钥；在 Windows 保持
`%APPDATA%\ValueCell\.env`。两份文件可以共享变量名，但变量值按机器网络环境分别填写。

## 1. 本机开发环境

本机后端读取：

```text
%APPDATA%\ValueCell\.env
```

Windows 示例：

```text
C:\Users\Administrator\AppData\Roaming\ValueCell\.env
```

本机访问 VPS PostgreSQL 时，使用 VPS 的公网 IP 或域名：

```env
VALUECELL_DATABASE_URL=postgresql+psycopg2://用户名:密码@VPS公网IP:5432/数据库名
```

本机行情优先 Gate：

```env
VALUECELL_MARKET_DATA_PROVIDERS=gate,okx,binance,mexc
VALUECELL_MARKET_REFRESH_S=3600
```

改完后重启本机后端。

## 2. VPS Docker 环境

VPS Docker Compose 读取：

```text
docker/runtime/.env
```

不要将这个文件提交到 Git。建议权限：

```bash
chmod 600 docker/runtime/.env
```

PostgreSQL 如果直接运行在 VPS 宿主机，而 ValueCell 后端运行在 Docker 容器中，使用：

```env
VALUECELL_DATABASE_URL=postgresql+psycopg2://用户名:密码@host.docker.internal:5432/数据库名
```

`docker-compose.local.yml` 已添加：

```yaml
extra_hosts:
  - "host.docker.internal:host-gateway"
```

不要让 VPS 容器通过 VPS 公网 IP 访问本机 PostgreSQL；这会绕远路，也可能被防火墙或公网监听规则阻断。

## 3. VPS PostgreSQL 必要配置

如果 PostgreSQL 只监听 `127.0.0.1`，Docker 容器无法访问。需要仅允许 Docker 网桥访问，不能开放到公网。

典型配置：

```conf
# postgresql.conf
listen_addresses = '127.0.0.1,172.17.0.1'
```

```conf
# pg_hba.conf
host    scholarx    scholarx    172.17.0.0/16    scram-sha-256
```

Docker 网桥实际网段可能不同。先在 VPS 查看：

```bash
docker network inspect bridge
```

然后用实际 Subnet 替换 `172.17.0.0/16`。

重载 PostgreSQL 后，在后端容器里验证：

```bash
docker compose -f docker-compose.local.yml exec backend python -c \
  "from valuecell.server.config.settings import get_settings; print(get_settings().DATABASE_URL)"
```

输出应包含：

```text
host.docker.internal:5432
```

## 4. 部署启动

```bash
docker compose -f docker-compose.local.yml config
docker compose -f docker-compose.local.yml up -d --build
docker compose -f docker-compose.local.yml ps
docker compose -f docker-compose.local.yml logs -f backend
```

Compose 中的后端和前端端口仅绑定到 `127.0.0.1`：

```text
后端：127.0.0.1:18000
前端：127.0.0.1:18080
```

外网访问请交给宿主机 Nginx/Caddy/Traefik 反向代理并配置 HTTPS。

### 4.0 Python / Node 构建网络策略

美国或欧洲 VPS 默认应使用官方 PyPI：

```text
https://pypi.org/simple
```

`docker-compose.local.yml` 已将 `UV_DEFAULT_INDEX` 默认值设为官方 PyPI。不要在美国
VPS 默认使用 `pypi.tuna.tsinghua.edu.cn`；它是跨境镜像，可能在 Docker BuildKit
网络中出现 wheel 下载超时。需要覆盖时，只能在确认容器构建网络可达后显式设置：

```bash
export UV_DEFAULT_INDEX=https://<已验证可达的镜像>/simple
docker compose -f docker-compose.local.yml build --no-cache backend
```

量化 SaaS 后端运行依赖不包含旧 Research Agent 的 SEC 工具链。
`edgartools` 及其 `unidecode` 依赖仅属于 `research` 可选组，因此：

```bash
uv sync --frozen --no-dev --no-install-project
```

不应下载 `edgartools` 或 `unidecode`。若构建日志仍下载它们，说明 VPS 使用了旧的
`python/uv.lock` 或旧 Docker 构建缓存；必须执行 `git pull` 后使用 `--no-cache` 重建。

后端 Dockerfile 仅安装锁定的第三方量化运行依赖，并通过 `PYTHONPATH=/app` 直接运行
`valuecell.server.main`。它不会在镜像构建阶段二次构建本项目 wheel，因此不会为
`hatchling` 额外发起网络下载。

前端仍使用项目锁定的 Bun `1.3.3`；不要为了网络下载问题改为 npm/yarn。Node 基础镜像
不可达时才将镜像前缀改为已验证可达的 registry mirror，保持 Node 版本 `22-alpine` 不变。

## 4.1 本机与 VPS 发布一致性检查

本机通过不等于 VPS 已通过。每次发布必须用同一份源码重新构建 VPS 镜像，并在
VPS 上重新执行同一条纸面策略验收。不要复制本机 `.env`；复制源码与模板即可。

### 两端必须一致的行为配置

两端都必须显式设置：

```env
VALUECELL_QUANT_ONLY_MODE=true
VALUECELL_MARKET_DATA_PROVIDERS=gate,okx,binance,mexc
VALUECELL_MARKET_DATA_MAX_CONCURRENT_FETCHES=12
VALUECELL_LIVE_TRADING_ENABLED=false
```

两端数据库地址不同。每个环境的前端与后端必须使用该环境自己的认证配置；本机和
VPS 不要求、也不建议共享 JWT 或加密主密钥：

| 配置 | Windows 本机 | VPS Docker |
| --- | --- | --- |
| `VALUECELL_DATABASE_URL` | `@VPS公网IP:5432/...` | `@host.docker.internal:5432/...` |
| `VALUECELL_JWT_SECRET` | 本机后端运行时 secret | VPS 后端运行时 secret |
| `VALUECELL_CREDENTIAL_MASTER_KEY` | 本机运行时 master key | VPS 运行时 master key |
| 前端 API 地址 | `http://localhost:8000/api/v1` | `/api/v1` |

JWT 和加密主密钥只在各自运行时 `.env` 填写，不能放进前端构建变量、Git 或 Docker
镜像。环境间不共享浏览器登录 token；切换环境后应重新登录。

### 4.2 HiCode AI 顾问配置

策略保存后的 AI 配置审阅复用后端既有的模型 provider 配置。当前 HiCode 通过
OpenAI-compatible provider 调用，不需要新增 token，也不会将 token 发送给浏览器、
数据库或任何策略记录。

在 VPS 的 `docker/runtime/.env`（或等价的密钥管理器）设置：

```env
PRIMARY_PROVIDER=openai-compatible
OPENAI_COMPATIBLE_BASE_URL=https://www.hi-code.cc/v1
OPENAI_COMPATIBLE_API_KEY=<HiCode proxy token>
```

模型 ID 仍由 `python/configs/providers/openai-compatible.yaml` 的 `default_model` 控制；
只有在 HiCode 账户确认支持其他模型时才修改该配置。启动后，
`POST /api/v1/rule-strategies/{strategy_id}/advisory-analysis` 使用同一代理和 token。

AI 输出仅为只读配置审阅：不能修改规则、不能影响 `buy/sell/no_op`、不能创建纸面或
真实订单。未设置有效 token 或模型不可用时，接口会返回 `503`，前端不会编造分析结果。

### 每次 VPS 发布的固定流程

```bash
cd /opt/valuecell
git pull
test -f docker/runtime/.env
docker compose -f docker-compose.local.yml config
docker compose -f docker-compose.local.yml up -d --build --force-recreate
docker compose -f docker-compose.local.yml ps
docker compose -f docker-compose.local.yml logs --tail=150 backend
curl --fail http://127.0.0.1:18000/api/v1/healthz
```

日志中不能出现以下错误；任一出现即停止发布并修复：

```text
ModuleNotFoundError: No module named 'agno'
null value in column "strategy_id" of relation "rule_strategies"
```

后者表示 VPS 镜像没有包含当前规则策略创建代码，必须重新构建，不能只重启旧容器。

### 纸面策略上线验收

在 VPS 前端登录后，新建一条最小纸面策略并填写 `Initial paper capital=1000`。
保存成功必须返回策略 ID，并在“Paper account”显示：

```text
Cash: 1000.00 USDT
Equity: 1000.00 USDT
Realized PnL: 0.00 USDT
Unrealized PnL: 0.00 USDT
```

随后访问：

```text
GET /api/v1/rule-strategies/<strategy_id>/account
```

应返回同一初始资金、现金和权益。该操作只创建纸面账户，不发送任何交易所订单。

### 本机开发固定启动方式

后端：

```bash
cd python
uv run python -m valuecell.server.main
```

前端：

```bash
cd frontend
set VITE_API_BASE_URL=http://localhost:8000/api/v1
npm run dev -- --host 127.0.0.1 --port 5173
```

使用 `http://127.0.0.1:5173/login` 登录。若页面显示 `Failed to fetch`，先确认
浏览器请求地址为 `http://localhost:8000/api/v1/...`，再确认后端 `healthz` 返回 200。

## 5. 市场数据基础层保证

市场数据由后端统一拥有，不由 Dashboard、Charts 或浏览器直接访问交易所。

```text
交易所多源
  -> 后端启动预热
  -> 后端默认市场快照（BTC/ETH/SOL，1h）
  -> Dashboard / Charts / 策略调度读取同一快照
```

运行时约定：

```env
VALUECELL_MARKET_DEFAULT_SYMBOLS=BTC-USDT,ETH-USDT,SOL-USDT
VALUECELL_MARKET_DEFAULT_INTERVAL=1h
VALUECELL_MARKET_DEFAULT_LOOKBACK=240
VALUECELL_MARKET_REFRESH_S=60
VALUECELL_MARKET_DATA_MAX_CONCURRENT_FETCHES=12
VALUECELL_MARKET_DATA_PROVIDER_ATTEMPTS=2
```

- 后端启动后立即异步预热；不阻塞健康、登录或策略 API。
- 每 60 秒刷新一次；页面刷新频率只决定读取后端快照的频率，不增加交易所请求。
- 每个交易对独立按 `gate -> okx -> binance -> mexc` 失败回退；每个 provider 可重试两次。
- 上游短暂失败时，只保留此前真实获取的 K 线，并在前端标记 `Data delayed`；不会生成模拟 K 线。
- 一个交易对失败不会清空其他交易对；策略只暂停缺少可用行情的该轮决策，纸面账户和历史权益不被重置。
- 若所有 provider 都失败且该进程从未取得过该交易对真实 K 线，接口仍会诚实返回不可用。此时不能可靠计算新的策略信号或收益。
- 每次成功刷新会将公开 K 线快照写入运行时目录的 `market_snapshot.json`；后端重启后先恢复该份此前真实数据，再后台刷新。该文件不含交易密钥或用户凭据。

默认快照可通过响应头确认：

```bash
curl -i "http://127.0.0.1:18000/api/v1/crypto-market/indicators?symbols=BTC-USDT&interval=1h&lookback=240"
```

应包含：

```text
X-ValueCell-Market-Cache: default-snapshot
```

这表示页面请求读取的是后端共享快照，不是一次新的交易所抓取。


### 5.1 行情图表验证

核心趋势图通过后端共享市场快照获取 K 线和指标，不依赖浏览器直接访问
TradingView/Binance。

VPS 验证：

```bash
curl http://127.0.0.1:18000/api/v1/healthz
curl http://127.0.0.1:18000/api/v1/crypto-market/health
curl -i "http://127.0.0.1:18000/api/v1/crypto-market/indicators?symbols=BTC-USDT&interval=1h&lookback=240"
```

最后一条应返回真实 `candles` 数组和：

```text
X-ValueCell-Market-Cache: default-snapshot
```

不要在默认 Dashboard/策略行情验证中传递 `providers=gate`，否则会绕过多源回退和
共享快照。

## 6. 数据源不可用的含义与排查

当页面显示“数据提供方暂无法提供市场数据，未显示图表”时，含义是后端未取得
任何真实 K 线，前端会刻意不伪造图表。当前本机日志已确认 Gate 返回：

```text
WinError 10054: 远程主机强迫关闭了一个现有的连接
```

这是数据源与当前网络出口之间的连接重置，不是登录、数据库或前端图表错误。

在部署目标主机上逐项执行：

```bash
curl -I --connect-timeout 10 https://api.gateio.ws/api/v4/spot/candlesticks
curl -I --connect-timeout 10 https://www.okx.com/api/v5/market/candles
curl -I --connect-timeout 10 https://api.binance.com/api/v3/klines
```

若任一请求被重置、超时或返回网络层错误，需要为该服务器配置可用的出站网络，或
选择该网络可访问的受支持数据源。不要用随机/模拟 K 线填充图表。

建议使用适度并发，避免后台策略批量抓取时阻塞前台行情接口，同时控制交易所限流风险：

```env
VALUECELL_MARKET_DATA_MAX_CONCURRENT_FETCHES=12
```

修改后重建后端容器：

```bash
docker compose -f docker-compose.local.yml up -d --force-recreate backend
```

如果 Gate 可访问但 Binance/OKX 不可访问，系统仍会使用 Gate 提供 K 线和指标。

## 7. 密钥安全

不要将数据库密码、JWT Secret、加密主密钥、交易所 Secret 写进：

```text
Git 仓库
Dockerfile
浏览器前端 .env
截图、聊天记录或日志
```

此前暴露过的数据库密码和模型 API Key 都应立即轮换。
