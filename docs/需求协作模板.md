# 需求协作模板

> 复制本模板填写。未知项写“待定”，不要猜测；我会先依据已有代码、配置和行业惯例补齐，
> 只有会改变产品取舍的问题才向你确认。

## 0. 提交信息

- **需求标题**：
- **需求类型**：新功能 / 改造 / 缺陷修复 / 性能 / 安全 / 界面 / 调研
- **优先级**：P0（阻塞）/ P1（本期）/ P2（可排期）
- **期望交付**：只出方案 / 方案后实现 / 直接实现 / 先做原型
- **期望完成标准**：

## 1. 业务目标

- **要解决的问题**：现状具体哪里不够用？
- **目标用户**：谁在什么场景下使用？个人客户、企业租户、平台管理员还是内部运营？
- **成功结果**：上线后用户能够完成什么，业务如何判断成功？
- **不做什么**：明确排除的范围，避免需求无限扩大。

## 2. 当前情况与背景

- **当前流程**：用编号写清楚用户现在如何操作。
- **已有页面 / 接口 / 模块**：路径、页面名称、截图或相关文件名。
- **现有痛点或错误**：包含复现步骤、实际结果、期望结果。
- **参考对象**：竞品、截图、原型、文档链接；说明希望借鉴什么，而不是只给链接。

## 3. 目标流程

用用户视角说明从开始到结束的完整流程：

1. 用户 A 在页面 / 接口 B 执行动作 C。
2. 系统校验 D，并展示 / 生成 E。
3. 成功时系统执行 F；失败时提示 G，用户可以 H。
4. 需要审批、定时任务、异步处理、通知或人工运营时，写明触发条件和责任人。

## 4. 功能清单与规则

### 4.1 必须交付

- [ ] 功能 1：输入、处理、输出。
- [ ] 功能 2：输入、处理、输出。

### 4.2 可选或后续交付

- [ ] 功能：为什么可以后置。

### 4.3 业务规则

- **默认值**：例如单笔纸面交易默认 100 USDT。
- **可配置项**：谁能配置、配置范围、何时生效、是否要审计。
- **校验与限制**：余额不足、重复提交、金额边界、时间边界、并发操作。
- **状态变化**：状态名称、可从什么状态变到什么状态、谁可操作。
- **异常处理**：接口失败、第三方不可用、数据不完整、权限不足时的行为。

## 5. 权限、租户与商业规则

没有涉及可写“无”；SaaS 需求应尽量填写。

- **可访问主体**：平台管理员 / 个人租户 / 企业租户 / 具体角色。
- **角色权限**：每个角色可查看、创建、修改、删除、审批什么。
- **租户隔离**：数据属于哪个租户；是否允许跨租户管理；切换租户后的行为。
- **账户开通规则**：订阅、合同、试用、到期、续费、停用后的可用范围。
- **计费或分成**：价格、币种、税费、周期、收益口径、高水位线、人工或自动结算。
- **审计要求**：必须记录的操作者、对象、前后值、原因、时间。

## 6. 页面与交互

- **涉及页面 / 菜单**：新增、修改、隐藏或保持不变的页面。
- **页面结构**：列表、筛选、详情、创建表单、审批、导出等。
- **关键交互**：加载、空状态、禁用状态、确认弹窗、成功提示、错误提示。
- **响应式与可访问性**：桌面优先 / 移动优先；键盘、读屏或多语言要求。
- **设计约束**：沿用现有设计系统，或提供截图 / Figma / 品牌要求。

## 7. 数据、接口与外部依赖

- **新增或变化的数据**：字段名称、类型、默认值、是否敏感、保留期限。
- **接口契约**：调用方、请求、响应、错误码、幂等要求。
- **外部系统**：交易所、支付、邮件、企业微信等；认证方式、沙箱、限流、回调。
- **迁移要求**：已有数据如何兼容、回滚方案、是否允许停机。

## 8. 安全与合规

- **敏感数据**：密码、API Key、交易凭据、个人信息、财务数据的处理要求。
- **禁止事项**：例如不保存交易所私钥、不向浏览器返回密钥、不记录敏感日志。
- **安全控制**：权限校验位置、二次确认、IP 白名单、审批、速率限制。
- **合规要求**：地域、审计、数据删除、合同或监管要求。

## 9. 验收标准

每条标准应可观察、可测试，建议用“给定 / 当 / 则”：

- [ ] 给定 ___，当用户 ___，则系统 ___。
- [ ] 给定余额不足，当策略触发买入，则不创建订单，并显示 / 记录 ___。
- [ ] 给定企业租户成员角色为 ___，当其访问 ___，则允许 / 拒绝 ___。
- [ ] 给定服务到期，当用户尝试 ___，则后端返回 ___，前端展示 ___。
- [ ] 给定平台管理员登录，当权限接口返回后，则显示平台控制台；普通租户始终不显示。

## 推荐协作方式

1. 先提交本模板的“业务目标、目标流程、必须交付、验收标准”。
2. 我先阅读现有实现，输出：影响范围、关键风险、实施计划、需你决定的少数产品取舍。
3. 你确认有分歧的取舍后，我完成实现、迁移、受影响调用方和必要验证。
4. 我交付时提供：改动文件、行为变化、验证证据、部署 / 配置动作，以及仍存在的风险。
5. 新需求继续引用本文件或已有需求编号；变更需求时写“变更原因、旧规则、新规则、影响时间”。

需求无需一次写得完美。优先写清楚“为什么做、谁使用、完成后必须发生什么、绝对不能发生什么”；
实现细节、字段和页面结构可以由我依据现有系统补全并与你确认。
